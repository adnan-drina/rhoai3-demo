# Stage 230 KFP Data Preparation

This directory contains the first Stage 230 KFP source for Dutch government
publication data preparation.

## Source Alignment

- Official product authority: [RHOAI 3.4 Prepare your data for AI consumption](https://docs.redhat.com/en/documentation/red_hat_openshift_ai_self-managed/3.4/html/customize_models_for_gen_ai_and_agentic_ai_applications/prepare-your-data-for-ai-consumption_custom-models)
- RAG implementation pattern: [Build an enterprise RAG system with OGX](https://developers.redhat.com/articles/2026/05/26/build-enterprise-rag-system-ogx)
- AG News reference implementation: [abdelhamidfg/agnews-rag-demo](https://github.com/abdelhamidfg/agnews-rag-demo)
- Docling KFP reference: [opendatahub-io/data-processing stable branch](https://github.com/opendatahub-io/data-processing/tree/stable/kubeflow-pipelines)

The first implementation adapts the `docling-standard` pattern for a single
public Staatsblad PDF. It does not use `docling-vlm` because the current
document is a normal text publication, not a scanned or image-heavy document.

## Files

| File | Purpose |
|------|---------|
| `dutch_publication_docling_pipeline.py` | KFP v2 pipeline definition for the single-document Docling preparation run |
| `components/dutch_docling_components.py` | Modular KFP components for S3 PDF download, Docling conversion, Stage 230 chunk generation, S3 output upload, and metrics |
| `../run-docling-pipeline.sh` | Repository runner that compiles the pipeline, imports a pipeline version into DSPA, starts a run, waits for completion, reviews S3 output, and records evidence |

## Compile

```bash
python stage-230-private-data-rag/kfp/dutch_publication_docling_pipeline.py \
  --output /tmp/stage-230-dutch-publication-docling.yaml
```

Compiled YAML is generated on demand. The source pipeline is committed, while
pipeline import, version creation, and run execution are handled by
`run-docling-pipeline.sh` against the GitOps-managed DSPA server.

## Run Through DSPA

After Stage 230 is deployed and the `dspa-enterprise-rag` route exists:

```bash
./stage-230-private-data-rag/run-docling-pipeline.sh
```

The runner reads the source PDF from the deterministic Stage 230 NooBaa bucket
key `raw/dutch-government/stb-2022-14.pdf`, writes the prepared JSONL artifact
to `processed/dutch-government/stb-2022-14-docling-kfp-chunks.jsonl`, reviews
the artifact in-cluster, and stores run evidence in
`enterprise-rag/stage230-docling-pipeline-evidence`.

The full stage gate also indexes that pipeline output through the same RAG
smoke helper:

```bash
RHOAI_STAGE230_RUN_DSPA_PIPELINE=true ./stage-230-private-data-rag/validate.sh
```

## Runtime Notes

The component uses the same Docling runtime image reference as the reviewed
`opendatahub-io/data-processing` stable branch:
`quay.io/fabianofranz/docling-ubi9:2.54.0`. Treat this as a reviewed demo
runtime dependency, not an operator-managed Red Hat product operand.

The pipeline uses the `data-processing-docling-pipeline` Secret contract for
S3 input and output:

- `S3_ENDPOINT_URL`
- `S3_ACCESS_KEY`
- `S3_SECRET_KEY`
- `S3_BUCKET`
- `S3_PREFIX`

Those values are generated by `deploy.sh` from the active project
ObjectBucketClaim. Do not commit credentials.

For this demo, the DSPA artifact store and `data-processing-docling-pipeline`
Secret use the in-cluster NooBaa HTTP service endpoint
`http://s3.openshift-storage.svc:80`. The dashboard/workbench S3 connection may
continue to use the OBC-advertised HTTPS endpoint. If the DSPA artifact store
endpoint is changed, recreate the DSPA; the generated workflow-controller
artifact repository is not a durable GitOps surface.
